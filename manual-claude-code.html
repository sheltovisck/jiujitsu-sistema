Claude Code para Iniciantes

Tudo que você precisa saber para começar a usar o Claude Code com eficiência — arquivos, comandos, organização e boas práticas.

⚡ O que é Claude Code

Claude Code é a CLI oficial da Anthropic que integra o modelo Claude diretamente no seu terminal, editor de código e fluxo de trabalho. Ele lê os arquivos do seu projeto, executa comandos, edita código e mantém memória entre sessões.

🖥️

CLI (Terminal)

Acesso via claude no terminal. Principal forma de uso, sem interface gráfica.

🖱️

Desktop App

Aplicativo nativo para Windows e Mac com interface visual integrada.

🔌

IDE Extensions

Plugins para VS Code e JetBrains — Claude dentro do editor.

🌐

Web App

Acesso via claude.ai/code no navegador, sem instalação.

Modelos Disponíveis (2026) Opus 4.7 · Sonnet 4.6 (padrão) · Haiku 4.5 — use /model para trocar. O Fast Mode usa Opus com saída mais rápida, ativado com /fast.

📦 Instalação

  1. Instale o Node.js 18+ (necessário para o pacote npm)
  2. Execute no terminal: 
    npm install -g @anthropic-ai/claude-codebash
  3. Autentique com sua conta Anthropic: 
    claude auth loginbash
  4. Inicie em qualquer diretório de projeto: 
    cd meu-projeto
claudebash
Dica de primeiro uso Na primeira sessão de um projeto novo, peça ao Claude para executar /init — ele vai ler o código e gerar um CLAUDE.md automaticamente.

📄 CLAUDE.md — O Arquivo Mais Importante

O CLAUDE.md é lido automaticamente em toda sessão. É onde você ensina ao Claude como funciona seu projeto — convenções, comandos de build, arquitetura, regras de estilo.

Onde colocar

projeto/
├── CLAUDE.md ← raiz do projeto (principal)
├── CLAUDE.local.md ← suas preferências pessoais (não commitar)
├── src/
│ └── CLAUDE.md ← regras específicas para src/
└── docs/
└── CLAUDE.md ← regras para documentação

Estrutura recomendada

# Nome do Projeto

## Visão Geral
Breve descrição do que o projeto faz e sua arquitetura.

## Comandos Essenciais
```bash
npm run dev      # servidor de desenvolvimento
npm run build    # build de produção
npm run test     # rodar testes
npm run lint     # verificar código
```

## Arquitetura
- src/components/  → componentes React reutilizáveis
- src/api/         → chamadas à API externa
- src/utils/       → funções utilitárias puras

## Convenções de Código
- TypeScript strict mode ativado
- Componentes em PascalCase, funções em camelCase
- Sem comentários desnecessários — nomes claros no código
- Testes unitários para toda lógica de negócio

## Variáveis de Ambiente
- DATABASE_URL → string de conexão PostgreSQL
- API_KEY       → chave da API externa (nunca commitar)

## O que NÃO fazer
- Não usar `any` no TypeScript
- Não commitar arquivos .env
- Não criar abstrações desnecessárias
markdown
CLAUDE.local.md não deve ir para o git Adicione CLAUDE.local.md ao .gitignore. Use-o para preferências pessoais que não precisam ser compartilhadas com o time.

⚙️ settings.json — Configuração

Controla permissões, hooks, variáveis de ambiente e comportamentos automáticos. Existe em três níveis, do mais específico ao mais geral:

Nível 1 — Projeto .claude/settings.json
Nível 2 — Local .claude/settings.local.json
Nível 3 — Usuário ~/.claude/settings.json

Exemplo de settings.json básico

{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(git log *)",
      "Read(**/*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)"
    ]
  },
  "env": {
    "NODE_ENV": "development"
  }
}
json

Hooks — Automações

Hooks executam comandos shell automaticamente em resposta a eventos do Claude Code. São configurados no settings.json.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo '[Hook] Executando comando bash...'"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --silent"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Sessão encerrada em: $(date)' >> .claude/sessoes.log"
          }
        ]
      }
    ]
  }
}
json
Use /fewer-permission-prompts O skill /fewer-permission-prompts analisa seu histórico e adiciona automaticamente ao settings.json os comandos que você permite com frequência, reduzindo interrupções.

🧠 Sistema de Memória

Claude Code mantém uma memória persistente entre sessões em ~/.claude/projects/[projeto]/memory/. Isso permite que ele lembre suas preferências, contexto do projeto e padrões de trabalho.

Tipos de Memória

👤

user

Seu papel, nível de experiência, preferências de comunicação.

💬

feedback

Correções e validações de abordagem — o que fazer e o que evitar.

🗂️

project

Contexto do projeto: decisões, metas, prazos, motivações.

🔗

reference

Onde encontrar informações externas (dashboards, tickets, docs).

Estrutura de um arquivo de memória

---
name: preferencias-codigo
description: Como o usuário prefere que o código seja escrito
metadata:
  type: feedback
---

Não adicionar comentários óbvios — o código deve ser autoexplicativo.

**Por quê:** O usuário considera comentários redundantes como ruído.
**Como aplicar:** Só comentar quando o "porquê" não está claro pelo código.
markdown

MEMORY.md — O índice

O arquivo MEMORY.md é o índice de todas as memórias. Ele é carregado automaticamente em toda sessão:

- [Perfil do usuário](user_profile.md) — desenvolvedor Python, iniciante em React
- [Preferências de código](feedback_code.md) — sem comentários, TypeScript strict
- [Contexto do projeto](project_context.md) — sistema de cadastro, prazo em julho
markdown
Pedindo ao Claude para lembrar algo Basta dizer: "Lembre-se de que prefiro funções pequenas e focadas" ou "Salve na memória que usamos PostgreSQL neste projeto".

🔒 Sistema de Permissões

Claude Code pede confirmação antes de executar ações potencialmente destrutivas. Você controla quais são aprovadas automaticamente.

Modos de Permissão

🔐

Default

Pede confirmação para ações que afetam arquivos ou executam comandos.

✅

Auto-approve

Aprova automaticamente ações seguras (leitura, comandos permitidos).

🚀

Yolo / Autônomo

Executa tudo sem pedir confirmação — use com cautela em projetos novos.

Padrões de permissão no settings.json

{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git log *)",
      "Bash(git diff *)",
      "Bash(npm run dev)",
      "Bash(npm run test)",
      "Bash(npm run lint)",
      "Read(**/*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Bash(DROP TABLE *)"
    ]
  }
}
json
Ações que sempre pedem confirmação Push para repositórios remotos, exclusão de arquivos, reset --hard, operações em banco de dados produção, e qualquer comando que afete sistemas externos.

✍️ Como Escrever Bons Prompts

Seja específico sobre o contexto

❌ Vago

"Arruma o bug no login"

✅ Específico

"Em src/auth/login.ts:47, a função validateToken retorna undefined quando o JWT expirou, mas deveria lançar TokenExpiredError. Corrija isso."

Tipos de pedido e como formular

Para correção de bugs

O teste em tests/auth.test.ts linha 23 está falhando com:
TypeError: Cannot read property 'id' of undefined

O erro ocorre em src/middleware/auth.ts:15 quando req.user
não está definido. Investigue a causa raiz e corrija.

Para novas funcionalidades

Adicione paginação ao endpoint GET /api/products.
- Parâmetros: ?page=1&limit=20 (padrão: page=1, limit=20)
- Retorno: { data: [...], total: number, page: number, pages: number }
- Siga o padrão dos outros endpoints em src/routes/
- Adicione testes em tests/routes/products.test.ts

Para refatoração

Em src/utils/helpers.ts, a função formatDate é chamada em
12 lugares diferentes com lógica duplicada. Extraia a lógica
comum para uma função reutilizável, mas não altere o comportamento.
Inclua localização exata Sempre que possível, inclua nome do arquivo e número de linha. Isso reduz o tempo que o Claude gasta procurando e aumenta a precisão da resposta.

Comandos úteis no prompt

FraseEfeito
"não explique o código"Resposta mais direta, sem explicações longas
"não adicione comentários"Claude não adiciona comentários no código
"só leia, não edite"Claude apenas analisa sem modificar
"liste os arquivos relevantes"Útil para explorar antes de editar
"faça um plano antes de executar"Claude apresenta a abordagem para aprovação

📊 Gerenciar Contexto

O contexto da conversa tem um limite. Sessões longas ficam lentas e mais caras. Gerencie proativamente.

Quando e como limpar

  • Use /clear ao trocar de tarefa completamente
  • Use /compact quando o contexto está grande mas a conversa ainda é relevante
  • O sistema comprime automaticamente quando necessário — o trabalho não é perdido

Boas práticas de contexto

  • Uma tarefa por sessão — evite misturar bugs, features e refatorações
  • Seja cirúrgico — "edite apenas o arquivo X" evita que Claude leia tudo
  • Use CLAUDE.md — não repita contexto a cada sessão; coloque no arquivo
  • Referência por caminho — "veja src/api/users.ts" é mais eficiente que colar o código
Cache de prompts O cache de prompts tem TTL de 5 minutos. Se sua sessão ficou inativa por mais de 5 minutos, o Claude paga o custo de reprocessar o contexto. Prefira sessões contínuas e ativas para economia.

🚀 Dicas Avançadas

Combine ferramentas para tarefas complexas

# Peça ao Claude para fazer tudo em sequência:
"1. Leia o schema em prisma/schema.prisma
 2. Crie o modelo User com os campos necessários
 3. Gere a migration com: npx prisma migrate dev --name add-user
 4. Adicione o endpoint POST /api/users em src/routes/users.ts
 5. Escreva os testes em tests/routes/users.test.ts"

Use o modo plan antes de grandes mudanças

"Antes de qualquer edição, liste todos os arquivos que serão
modificados e descreva as mudanças. Aguarde minha confirmação."

Subagentes para tarefas paralelas

"Execute duas buscas em paralelo:
1. Encontre todos os usos de `deprecated_function` no código
2. Verifique se há testes cobrindo os arquivos que a usam"

Integração com git

# Peça para fazer commits bem formatados:
"Faça commit das mudanças com uma mensagem no formato Conventional Commits"

# Revisão antes de push:
"Revise o diff atual e diga se está pronto para PR"

MCP Servers — Expandir Capacidades

Model Context Protocol (MCP) permite conectar Claude Code a ferramentas externas como Slack, Linear, GitHub, banco de dados, e mais.

# Adicionar um MCP server ao settings.json:

    "github": {
      "command": "npx",
      "args": ["@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}

⚠️ Erros Comuns e Como Evitar

1. CLAUDE.md vazio ou ausente

Problema Sem CLAUDE.md, o Claude não conhece as convenções do projeto e pode gerar código fora do padrão.

Solução: Execute /init no início de todo projeto novo. Atualize o CLAUDE.md quando a arquitetura mudar.

2. Dar tarefas grandes demais de uma vez

Problema "Refatore todo o sistema de auth" pode gerar mudanças incorretas em cascata.

Solução: Quebre em partes menores. "Primeiro só o middleware, depois os controllers, depois os testes."

3. Não verificar as mudanças

Problema Confiar que o Claude sempre acerta sem revisar as edições.

Solução: Sempre faça git diff antes de commitar. Use /verify para confirmar que features funcionam.

4. Commitar segredos

Problema O Claude pode sugerir usar variáveis de ambiente mas você commita o .env por acidente.

Solução: Sempre tenha .env no .gitignore. Nunca cole credenciais reais no chat com Claude.

5. Sessão longa sem gerenciar contexto

Problema Contexto gigante aumenta custos e reduz precisão das respostas.

Solução: Use /compact regularmente. Use /clear ao trocar de tarefa.

6. Ignorar o sistema de permissões

Problema Aprovar tudo sem ler pode deixar o Claude executar comandos destrutivos.

Solução: Leia sempre o que está sendo executado. Configure o deny no settings.json para bloquear comandos perigosos.
